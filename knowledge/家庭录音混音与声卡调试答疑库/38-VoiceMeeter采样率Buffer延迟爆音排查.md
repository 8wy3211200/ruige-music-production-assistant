# VoiceMeeter采样率Buffer延迟爆音排查

适用问题：
- 返听延迟太大
- 爆音、卡顿、断续、机器人声
- 改了 Buffer 之后更不稳定
- Discord、会议软件、直播软件和 VoiceMeeter 采样率对不上

核心结论：
- VoiceMeeter 的全局延迟和稳定性，首先看 `A1` 选了什么设备和驱动类型。
- 不要一味追求最小 Buffer。对真实用户来说，稳定比纸面低延迟更重要。

驱动类型先怎么选：
- `ASIO`：如果你的物理声卡自带稳定 ASIO 驱动，优先用它做 `A1`。
- `WDM`：官方手册把它视为 Windows 下更适合低延迟和较好性能的常用选择。
- `KS`：也可能很低延迟，但不是所有设备都支持得好。
- `MME`：兼容性高，但延迟可能接近 `100ms`，更适合兜底，不适合追求实时返听。

先搞懂一件关键事：
- `A1` 决定了 VoiceMeeter 的主采样率、主 Buffer，也影响虚拟 ASIO 的配置。
- 所以“延迟大不大、稳不稳定”不要先去乱改每个软件，先看 `A1`。

更稳的实操顺序：
1. 先用最好的物理输出设备做 `A1`。
2. 先用默认值跑通，不要一上来就极限低延迟。
3. 需要返听时，优先试 `ASIO` 或 `WDM`。
4. 如果延迟还能接受，就不要继续盲目往下压 Buffer。

官方手册里对 Buffer 的有用边界：
- `MME` 常见缓冲区间大约在 `512` 到 `2048` samples。
- `WDM` / `KS` 可以低到 `256` samples，但不是所有电脑和设备都稳定。
- 默认值之所以更保守，是为了覆盖更多普通电脑配置。

什么时候该把 Buffer 调大：
- 出现爆音、断续、噼啪声。
- 声音像被切碎，或者偶发机器人声。
- 一开直播、会议软件或浏览器后就不稳定。

什么时候不要继续降：
- 一降到很小，电平表开始抖、声音开始断。
- 你已经不是“延迟大”，而是“系统开始不稳”。
- 这时继续压 Buffer，只会把“可用”变成“完全不能用”。

采样率怎么排查更实际：
- 项目里统一一个采样率，比追求更高更重要。
- 如果某些语音软件表现异常，官方手册特别提到像 `Discord` 这类应用可能更适合 `48 kHz`。
- 所以遇到“只在某个软件里炸裂或失真”，先看它吃的是不是 `48 kHz`。

VAIO 内部延迟不要乱改：
- 手册建议 VAIO 内部延迟不要低于当前相关 Buffer 时间的约 `3` 倍。
- `7168 samples` 是更稳的安全值，`3072` 或更小只适合在确认稳定后再试。
- 内部延迟压太低，也会造成断流、静音或各种不连续问题。

额外排查点：
- Windows 11 的电源节流、应用最小化后节能，有时会让音频流不稳定。
- 改过 Buffer、SWIFT 之类实验选项后更糟，先退回默认再判断。

- `VB-Audio Voicemeeter Standard User Manual` v1.1.2.2（DEC 2025）

一句话总结：
VoiceMeeter 的延迟优化从来不是“把 Buffer 拉到最小”，而是先把 `A1` 选对，再在稳定前提下逐步试低。
